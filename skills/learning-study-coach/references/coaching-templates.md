## 12. 命令讲解规范

- 每条关键命令都解释：命令名、参数、作用、风险。
- 涉及安装步骤时，说明每个工具“为什么需要它”。
- 涉及目录结构时，说明每个目录“放什么内容、后续如何用”。
- 涉及首次出现的术语时，先给白话解释，再补充技术定义。
- 教程省略前置知识时，要主动补讲，而不是等学习者追问。
- 根据讲解档位调整解释粒度，但不要省略成功信号和失败排查。

## 13. Python 初学教程示例

```markdown
### Step 1 - 运行第一段 Python 代码

#### 0) 新章节导学卡（固定 7 项）
- 本章一句话总览：先确认 Python 环境可运行，再进入语法学习。
- 本章提纲挈领：
  - 验证 Python 命令是否可用
  - 跑通第一条输出语句
  - 认识解释器交互模式
  - 为变量和控制流学习做准备
- 学习目标：
  - 能在终端启动 Python 解释器
  - 能执行 `print("hello")` 并看懂输出
- 需要掌握的内容：
  - 概念：解释器、终端、输出
  - 命令/操作：`python3`、`print(...)`
  - 检查点：能稳定复现输出
- 学习作用：
  - 以后所有 Python 练习都依赖这一步
  - 出现错误时能快速判断是环境问题还是语法问题
- 验收标准：
  - 可进入 Python 提示符
  - 可看到正确输出 `hello`
- 本章第一小步（只做这个）：
  - 动作：启动 `python3` 并执行 `print("hello")`
  - 完成后回传：终端输出 + 你对 `print` 的一句话理解

#### 1) 当前讲解档位
零基础档

#### 2) 当前所在位置
Python 入门教程 / 第 1 节 / 第 1 步

#### 3) 本步目标
确认你的电脑可以运行最简单的 Python 代码。

#### 4) 对应档位的讲解
这一步不是在学复杂语法，而是在确认“写出来的 Python 代码真的能跑起来”。如果这一步不通，后面所有练习都会卡住。

#### 5) 新词和隐藏前置知识
- Python：一种编程语言
- 终端：你输入命令、让电脑执行操作的地方
- print：让程序把内容显示出来

#### 6) 为什么先做这步
因为先确认运行环境正常，后面才值得继续学变量、条件、循环。

#### 7) 技术细节说明
- `python3`：启动 Python 解释器
- `print("hello")`：把 hello 输出到屏幕上
- 预期结果：屏幕看到 hello

#### 8) 命令/代码预讲（执行前）
- 命令族是什么：`python3` 是 Python 解释器启动命令。
- 当前子命令做什么：`print(...)` 是内置函数，用来输出文本。
- 参数格式怎么读：`print("hello")` 里 `"hello"` 是字符串参数。
- 为什么现在用它：先验证“代码能跑且有输出”，后续学习才有基础。
- 可扩展知识（同类命令或常见变体）：可用 `python3 --version` 先看版本；退出解释器可用 `exit()`。

#### 9) 现在请你执行
python3
print("hello")

#### 10) 成功信号
- 进入 Python 提示符
- 屏幕输出 `hello`

#### 11) 失败先排查
- 输入 `python3` 后是否提示 command not found
- 是否把中文引号写成了英文引号

#### 12) 完成后回传
- 终端输出
- 用一句话说说 `print` 是干什么的
```

## 14. Git 基础教程示例

```markdown
### Step 1 - 理解 git status

#### 0) 新章节导学卡（固定 7 项）
- 本章一句话总览：先学会读仓库状态，再做 add/commit 等操作。
- 本章提纲挈领：
  - 认识工作区、暂存区、提交记录
  - 学会用 `git status` 看状态
  - 区分 modified / untracked / staged
  - 为后续提交流程打基础
- 学习目标：
  - 能独立看懂 `git status` 输出
  - 能解释当前仓库“下一步该做什么”
- 需要掌握的内容：
  - 概念：工作区、暂存区、提交
  - 命令：`git status`
  - 检查点：看到状态后能判断是否需要 `git add`
- 学习作用：
  - 避免“没看状态就误操作”
  - 后续所有 Git 操作都依赖状态判断
- 验收标准：
  - 能指出当前分支和文件状态
  - 能解释 modified 与 untracked 差异
- 本章第一小步（只做这个）：
  - 动作：执行 `git status`
  - 完成后回传：命令输出 + modified/untracked 区别

#### 1) 当前讲解档位
夯实基础档

#### 2) 当前所在位置
Git 入门教程 / 第 2 节 / 工作区检查

#### 3) 本步目标
学会用 `git status` 看懂当前仓库处于什么状态。

#### 4) 对应档位的讲解
`git status` 像仓库的体检报告。它不会改代码，只会告诉你：哪些文件改了、哪些还没提交、哪些是新文件。

#### 5) 新词和隐藏前置知识
- 工作区：你当前正在修改的文件
- 暂存区：准备进入下一次提交的文件集合
- 提交：一次带说明的保存记录

#### 6) 为什么先做这步
因为 Git 的很多操作都依赖你先看懂当前状态，不然很容易在没搞清楚的情况下乱提交。

#### 7) 技术细节说明
- `git status`：查看仓库状态
- 不会修改任何文件
- 重点看 modified、untracked、staged 这几类提示

#### 8) 命令/代码预讲（执行前）
- 命令族是什么：`git` 是版本控制命令工具。
- 当前子命令做什么：`status` 用于展示仓库当前状态快照。
- 参数格式怎么读：本步用基础形式 `git status`，不带额外参数。
- 为什么现在用它：不先读状态，后续 add/commit 容易做错。
- 可扩展知识（同类命令或常见变体）：`git status -s` 可看精简输出；`git branch` 可看分支。

#### 9) 现在请你执行
git status

#### 10) 成功信号
- 看到当前分支名
- 看到工作区和暂存区状态说明

#### 11) 失败先排查
- 当前目录是不是 Git 仓库
- 是否提示 `not a git repository`

#### 12) 完成后回传
- `git status` 输出
- 你理解的 modified 和 untracked 区别
```

```markdown
  - 两条命令都可成功执行
  - 能解释 `source` 在本章的作用
- 本章第一小步（只做这个）：
  - 动作：先执行 `source /opt/ros/jazzy/setup.bash`

#### 1) 当前讲解档位

#### 2) 当前所在位置
ROS 2 Day1 / 环境确认 / 第 1 步

#### 3) 本步目标
确认 ROS 2 基础命令可用，再进入 workspace 练习。

#### 4) 对应档位的讲解
这一步就像先确认工具箱已经装好，避免后面练习时才发现基础工具根本不能用。

#### 5) 新词和隐藏前置知识
- ROS 2：机器人软件开发框架
- `source`：把某些环境设置加载到当前终端
- topic：ROS 2 里不同节点交换消息的通道

#### 6) 为什么先做这步
如果环境没有加载成功，后面创建包、运行节点、查看话题都会失败。

#### 7) 技术细节说明
- `source /opt/ros/jazzy/setup.bash`：把 ROS 2 Jazzy 的环境变量加载到当前 shell
- `ros2 topic list`：查看当前系统里可见的话题列表

#### 8) 命令/代码预讲（执行前）
- 命令族是什么：`ros2` 是 ROS 2 的主命令入口。
- 当前子命令做什么：`topic list` 用来查看当前可见话题。
- 参数格式怎么读：`source <setup.bash 路径>` 先加载环境，再执行 `ros2 topic list`。
- 为什么现在用它：先确认环境，再做后续节点练习，能减少排错成本。
- 可扩展知识（同类命令或常见变体）：可用 `ros2 node list` 查看节点；每次新终端通常都要重新 `source`。

#### 9) 现在请你执行
source /opt/ros/jazzy/setup.bash
ros2 topic list

#### 10) 成功信号
- 命令可执行且无 `command not found`
- topic 列表可以正常返回

#### 11) 失败先排查
- 路径 `/opt/ros/jazzy/setup.bash` 是否存在
- 是否把命令放在同一个终端里执行

#### 12) 完成后回传

- 你理解的 `source` 在这里的作用
```

```markdown
- `show`：把某个具体接口的字段结构打印出来。
- `example_interfaces/srv/AddTwoInts`：目标接口名，格式是 `包名/类型/接口名`。

#### 可扩展知识（先知道一点就够）
- `ros2 interface list`：先列出可用接口
- `ros2 interface package <包名>`：看某个包里有哪些接口
- `msg / srv / action` 三类接口在用途上不同，后续会分别遇到
```

```markdown
额外要求：
- 用表格展示“目标-完成率-证据”
- 明确标注“需要补证据”的结论
- 输出末尾给出 2 个自检问题，帮助我判断复盘是否落地
~~~
- 进一步改进点：
  - 强制“事实→结论”顺序，降低臆测风险
  - 加入优先级与执行条件，提升可操作性
  - 增加证据链要求，提升复盘可信度

#### 对比复盘（示例）
- 输出质量变化：从“流水账总结”变成“带证据和行动项的复盘单”
- 稳定性变化：多次输入不同素材后，结构仍保持一致
- 还可继续优化的点：加入“上周动作完成率追踪”字段形成闭环
```
